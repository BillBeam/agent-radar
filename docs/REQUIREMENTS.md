# Agent Radar — 需求书 & 协作模式

> 本文用第一人称记录**我（项目发起者）对这个 agent 的完整需求**，以及**我们三方（我 + Claude Code + 网页版 Claude）协作开发的方式**。
> 技术细节与实现现状见 [SPEC.md](SPEC.md)（唯一技术真相源）；这份是「我们想要什么 + 怎么一起造」。

---

## 一、这个 agent 是什么、我为什么要它

我是一名专注 **agent / harness 方向**的资深工程师。痛点：我想持续追踪这个领域**最前沿的工程级**技术（像读 Claude Code 源码那种深度），但不知道每天去哪找、而且英文原生前沿很多、我读英文偏慢。

所以我要做 **Agent Radar**：每天自动抓取全球最前沿的 agent/harness 工程内容，用 AI 理解后产出**中文详解**，按我的口味推送到我手上——并且它本身也在不断变强（有记忆、能对话、会自我进化）。它本身就是一个有分量的 agent harness 工程作品。

## 二、它最终要做到的四件事

1. **每天推送 + 中文详解**：把英文前沿理解透，产出中文详解（不是翻译，是讲懂），按我的口味推到钉钉。
2. **有记忆**：记得过往推送（同主题串成 thread 追踪进展），也记得我这个人（背景、知识水平、口味、反馈）。
3. **能对话讨论**：每天就推送的内容跟我讲解、讨论、答疑，基于今日事实 + 过往记忆。
4. **会自我进化**：对话中按上下文自动迭代系统本身（改源/权重/prompt）、提取记忆、创建/编辑工具与 skill；并把**每天读到的前沿 agent 技术，eval 验证后用来升级它自己**（它调研的就是 agent 技术，这些技术本就能改进它自身）。

## 三、硬性约束（每一条都不能违背）

1. **技术真实、可 defend**：实现要前沿且真实，每个设计决策的「为什么这么选 / 什么时候不用」都要想清楚、能 defend。不堆砌概念，要真正落地、对需求有用。
2. **走订阅、不额外计费**：LLM 调用走 `claude -p` headless + Claude Code 订阅额度，绝不引入按量计费的付费 API；嵌入用本地模型。
3. **钉钉为主 + 本地兜底**：推送主渠道是钉钉群机器人（加签）；本地 Markdown 归档兜底。渠道可插拔。
4. **每日精简 + 每周深读**：工作日推精简版（前沿够多才约 10 条），周末做一篇深度。
5. **中文详解要讲懂**：对标技术博客精读 / 源码笔记的深度，关键英文术语首次出现即解释；严格基于原文、反幻觉。
6. **质量把关、宁缺毋滥、实时前沿**：只推最前沿、实时、有工程深度的内容；前沿少的日子就少推，绝不用旧闻/营销稿/水文凑数。
7. **可拓展/可迭代/自进化拉满**：加一个能力 = 加一个文件或一行配置，**绝不牵一发动全身、缝缝补补**。
8. **用最前沿的技术、真实落地**：实现这个 agent 本身要用业界最前沿的 agent/harness 技术（参考 Claude Code 源码、本地参考 harness、以及能调研到的最新前沿）；不停留在概念，要真正实现、对需求真有用。
9. **边实现边调研（JIT）**：不在规划阶段一次性把调研做完。每个组件开工前先针对它做一次「当下 SOTA 调研 + 选型」（读源码 + web 最新），再写代码。
10. **生产级健壮、不是 demo**：虽然跑在本地，但要是一个完善健壮的 agent harness（容错/可观测/测试/数据完整性/可运维），不是玩具 demo。
11. **对话驱动开发、不派手动任务**：这个 agent 全程通过对话来实现，由 Claude Code 做一切执行（写码/跑/测试/调度）。我（用户）唯一需要在对话里给的：① 钉钉 webhook（只有我账号能建）② 口味校准 ③ 对自我修改的 diff 拍板「上 / 再改」。

## 四、关注的能力域（决定选题口味，也是 agent 自身要体现的）

抓取打分要偏向这些 agent 工程主题（也是 agent 自身架构尽量逐个命中的能力点）：Agent 编排与架构（任务分解、规划 CoT/ToT、多步推理）、多智能体协作（通信协议、角色分工、冲突消解、共识）、Reflection / Tool Use / Human-in-the-loop、LLMOps（部署、灰度、监控、Token 预算）、可观测性（全链路 Trace、幻觉检测、异常自动恢复）、Eval 框架与「数据-训练-评测」闭环（含 Hard Negative Mining）、RAG（混合检索、重排、知识图谱、上下文窗口管理）、工具接入层（MCP / Function Calling、参数校验与执行安全）、记忆系统（FTS5 + USER.md + LLM 选择，非向量；短期上下文 + 长期个性化记忆）、容器化与隔离。

## 五、运行机制

本地 `launchd` 定时唤醒 → `python -m radar --mode daily`：确定性步骤（抓取/去重/质量门/记忆/投递）纯 Python 跑；需要「智能」的步骤经 LLM 接口 → `claude -p`。因为没设 `ANTHROPIC_API_KEY` 且是订阅登录，这些调用走订阅额度、不按 API 计费。

## 六、我们怎么一起开发（三方协作模式）★

这个 agent 由**三方协作**持续开发：

```
        ┌──────────────────────────────────────────────────────────┐
        │                       我（项目发起者）                       │
        │  定方向 · 提需求 · 做决策（口味校准 / 自我修改 diff 拍板）      │
        └───────────────┬───────────────────────────┬──────────────┘
                        │ 发提示词                      │ 给反馈/决策
                        ▼                              ▼
        ┌──────────────────────────┐      ┌──────────────────────────┐
        │   Claude Code（实现者）    │◀────▶│  网页版 Claude（审查者）    │
        │  写码/跑/测试/commit/push  │ 代码  │  审查代码 + 设计我发给     │
        │  做一切执行                │ 审查  │  Claude Code 的提示词       │
        └──────────────────────────┘      └──────────────────────────┘
```

- **我（用户）**：把控方向、提出需求、做关键决策（口味校准、对自我修改的 diff 拍板「上/再改」）。不写代码、不跑命令。
- **Claude Code（本地，实现者）**：做**一切执行**——写代码、跑、测试、commit、push。每次外部行为有变化都把证据贴给我看。遵守上面第三节所有约束。
- **网页版 Claude（claude.ai，连接本公开仓库，审查者）**：两个职责——
  1. **审查 Claude Code 写的代码**：读这个仓库，从架构、正确性、健壮性、可维护性、是否符合 SPEC 与约束等角度提审查意见。
  2. **设计/打磨我发给 Claude Code 的提示词**：把我的模糊想法，结合对仓库现状的理解，转成结构清晰、信息充分、可直接执行的提示词，让 Claude Code 高质量落地。

**典型协作回路**：我有个想法 → 网页版 Claude 把它打磨成一条好提示词 → 我发给 Claude Code → Claude Code 实现并 push → 网页版 Claude 审查这次改动、提意见 → 我把意见带回给 Claude Code 修正。如此迭代。

> 给网页版 Claude：你既是**代码审查者**也是**提示词设计师**。审查时请具体到文件/函数、对照 SPEC 与约束、指出风险与更优解；设计提示词时请充分利用你对仓库现状的理解，让提示词可直接执行、边界清晰。先读 [SPEC.md](SPEC.md) 建立全局认知。
