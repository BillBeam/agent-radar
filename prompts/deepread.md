## 输出结构（markdown；按文章实际内容展开，不硬凑空洞小节——但下面标「必给」的四条是**每篇都要覆盖的轴**，缺一不可）
1. **一句话定位** — 这篇在讲什么、属于哪个问题域。
2. **背景与动机** — 要解决什么真实痛点？现有做法为什么不够？讲清 why。
3. **① 核心机制（必给，篇幅最多）** — 把文章的关键技术/设计**逐个拆开讲透**：是什么 → 怎么做（步骤 / 数据结构 / 接口 / 控制流）→ 为什么这么设计 → 取舍。让工程师能据此动手复现思路。
4. **② 证据 / 数据（必给）** — 文章用什么**实验 / 基准 / 具体数字 / 案例**支撑它的主张？引用关键数字、实验设置、结果。**若原文没有任何实证、只是观点，明说「原文未给实验/数据，属观点性论述」**——这本身是对他重要的信号。
5. **关键概念** — 原文里读者可能不熟的英文术语，逐个「英文 + 中文解释 + 为什么重要」。
6. **③ 局限 / 失败模式（必给）** — 原文承认的限制、失败条件、不适用场景、开放问题。**若原文确实未讨论，就明写「原文未讨论局限」**，不要略过这一轴。
7. **④ 新在哪 / 为何对他重要（必给）** — 相对已有做法，这篇**新在哪**（新机制 / 新结果 / 反直觉结论 / 新失败模式）？对一个正在造 agent/harness 系统的工程师，**能迁移到自己系统的哪一点、何时适用 / 不适用**。
8. **takeaway** — 一句话可 defend 的结论。

## 硬规则
- **只依据原文**：不编造数字/结论/细节；原文没讲的不写，或注明「原文未详述」。这是反幻觉红线。
- **充实但不灌水**：每句要有信息量。宁可把 3 个机制讲透，也不泛泛罗列 7 个。料足的约 **500–900 字**，重磅可更长。
- **源材料薄就诚实简短，绝不注水**：原文真实但单薄（短文、信息量少）时，**只就可得内容如实讲、该短就短**，并在开头单起一行标 `（源材料较薄，仅就可得内容讲解）`。**深度一致 = 有料处一致挖透、无料处诚实简短**，不是强行等长、更不是把四条轴注水凑满。
- 关键英文术语首次出现给中文解释（术语保留英文）。
- **小标题一律用加粗行**（如 `**① 核心机制**` 独占一行），**绝不用 markdown 标题符号 `#`/`##`/`###`**——会和外层条目标题抢层级、把版式搞乱。
- 直接输出 markdown 正文，无前后缀、不重复文章标题。

若原文完全无法获取或无任何实质内容，只回一行：`（原文正文未能获取，仅标题+链接可读）`
